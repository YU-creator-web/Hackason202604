1Application Hosted Media Bot とは

🎯 この実装を学ぶ目的

Teams 会議に Bot としてリアルタイム参加し、参加者の音声を受信・文字起こし・AI 応答できるアーキテクチャを理解します。会社の実装と同じ構成です。

📐 音声の流れ（アーキテクチャ）

Teams 会議（参加者が発話） │ │ SRTP / UDP（暗号化音声ストリーム） ▼ Media SDK（Windows DLL） │ AudioSocket.AudioMediaReceived（20ms ごとに発火） ▼ CallHandler.OnAudioMediaReceived() │ Marshal.Copy → byte[] に変換（640 バイト / 20ms） ▼ PushAudioInputStream.Write(data) │ PCM 16kHz 16bit mono ▼ Azure Speech SDK（SpeechRecognizer） │ Recognized イベント → 確定テキスト ▼ GPT-4o（Azure OpenAI）→ 応答テキスト生成 ▼ （本番）AudioSocket で Teams に音声送信

🆚 ACS Call Automation との違い

比較項目Application Hosted Media BotACS Call Automation
参加方法Graph Communications SDK 経由で Teams 会議に参加ACS 経由でコールに参加
音声処理ボット自身が音声を受信・処理（Application Hosted）Azure のクラウド側で処理
言語C#（.NET Framework 4.8）のみPython / C# / Java 等
OSWindows 必須（Media SDK が Windows DLL）どこでも動く
柔軟性音声バイト列に直接アクセス可能（高い）Azure 側の処理に依存（低い）

⚠️ なぜ .NET Framework 4.8 / Windows 必須なのか

Microsoft.Graph.Communications.Calls.Media SDK は内部的に Windows 専用のネイティブ DLL（Microsoft.Skype.Bots.Media.dll 等）を使用しています。このため：

  • .NET Framework 4.8（net48）が必須 — .NET 8 や Linux では動作しない
  • Windows 環境が必須 — Azure Linux App Service や AWS Linux では動作しない
  • 会社の実装が Azure Windows VM を使っている理由がここにある
💡 Web サーバーは ASP.NET Core（.NET 8 用）ではなく、OWIN（Open Web Interface for .NET）を使います。OWIN は .NET Framework 時代の Web ホスティング基盤です。

2.NET SDK のインストールと MediaBot プロジェクトの作成

📋 .NET SDK のインストール確認

ターミナルで以下を実行してインストール済みか確認します：

dotnet --version   # 8.x.x 以上が表示されればOK

インストールされていない場合は dotnet.microsoft.com/download から .NET SDK をダウンロードしてインストールしてください。

💡 .NET SDK 自体は .NET 8 でも、プロジェクトのターゲットフレームワークは net48（.NET Framework 4.8）を指定します。SDK は複数バージョン向けのビルドができます。

📋 プロジェクトの作成

リポジトリのルートで以下を実行します：

mkdir MediaBot
cd MediaBot
dotnet new console

生成された MediaBot.csproj と Program.cs が出発点です。

📋 csproj の設定

生成された MediaBot.csproj を以下の内容に書き換えます：

<Project Sdk="Microsoft.NET.Sdk">
  <PropertyGroup>
    <TargetFramework>net48</TargetFramework>
    <OutputType>Exe</OutputType>
    <Nullable>enable</Nullable>
    <ImplicitUsings>enable</ImplicitUsings>
    <LangVersion>latest</LangVersion>
    <Platforms>x64</Platforms>
    <PlatformTarget>x64</PlatformTarget>
    <AllowUnsafeBlocks>true</AllowUnsafeBlocks>
  </PropertyGroup>
</Project>
設定理由
net48Media SDK が .NET Framework 4.8 専用
PlatformTarget x64Media SDK のネイティブ DLL が x64 専用。これがないと BadImageFormatException が発生
AllowUnsafeBlocks音声バッファのアンマネージドメモリ操作に必要
LangVersion latestnet48 では C# 7.3 がデフォルト。nullable など最新構文を使うために必要

4ソースコードの実装（7ファイル）

📁 ファイル構成

MediaBot/ ├── MediaBot.csproj ├── appsettings.json # 設定ファイル（キーは Git に上げない） ├── Program.cs # エントリポイント・起動モード切替 ├── Startup.cs # OWIN Web サーバー設定 ├── SimpleDependencyResolver.cs # 手動 DI（.NET Framework 用） ├── Models/ │ └── AppSettings.cs # 設定クラス + JSON ローダー ├── Bot/ │ ├── BotService.cs # Graph Communications クライアント管理 │ ├── AuthenticationProvider.cs # Azure AD トークン取得（MSAL） │ └── CallHandler.cs # 1通話の音声受信・STT 処理 ├── Controllers/ │ └── PlatformCallController.cs # Teams Webhook の受け口 └── Services/ ├── TranscriptionService.cs # STT セッション管理 ├── SpeechTest.cs # マイクテスト用 └── ConversationTest.cs # 双方向会話テスト用

📄 各ファイルの役割

ファイル役割・ポイント
BotService.cs CommunicationsClientBuilder でクライアントを初期化。着信コールを受けて CallHandler を生成。 CreateMediaSession() は拡張メソッドで、第3引数 null の型曖昧性を IEnumerable<VideoSocketSettings>? noVideo = null で解消する必要がある。
AuthenticationProvider.cs IRequestAuthenticationProvider を実装。MSAL の ConfidentialClientApplicationBuilder でクライアント資格情報フローによりトークン取得。 ValidateInboundRequestAsync の戻り値は Task<RequestValidationResult>（Task ではない）。
CallHandler.cs AudioSocket.AudioMediaReceived に登録した OnAudioMediaReceived が 20ms ごとに呼ばれる。 Marshal.Copy(buffer.Data, data, 0, length) で IntPtr からバイト列に変換（unsafe 不要）。 バッファは必ず args.Buffer.Dispose() する（アンマネージドメモリのため GC が回収しない）。
TranscriptionService.cs PushAudioInputStream を使って PCM データを受け取る Speech セッションを作成。 音声フォーマット: 16kHz / 16bit / モノラル（Teams の出力形式に合わせる）。
ConversationTest.cs Teams なしで STT → GPT-4o → TTS のフルループをローカルテストするクラス。 Azure OpenAI は追加 SDK なし、HttpClient の REST 呼び出しで実装。

📄 appsettings.json の構造

{
  "AppSettings": {
    "AzureAdTenantId":        "（Azure AD テナント ID）",
    "AzureAdAppId":           "（アプリ登録のアプリケーション ID）",
    "AzureAdAppSecret":       "（クライアントシークレット）",
    "BotBaseUrl":             "（Azure VM の HTTPS URL または Dev Tunnel URL）",
    "SpeechKey":              "（Azure Speech Services のキー）",
    "SpeechRegion":           "japaneast",
    "AzureOpenAiKey":         "（Azure OpenAI / Foundry のキー）",
    "AzureOpenAiEndpoint":    "https://（リソース名）.openai.azure.com/",
    "AzureOpenAiDeployment":  "gpt-4o"
  }
}
⚠️ appsettings.json はキーを含むため、.gitignore で除外してください。

5Azure Speech Services リソースの作成

🎯 この手順の目的

Teams の音声（PCM 16kHz）をリアルタイムでテキストに変換するために Azure Speech Services を使います。STT（Speech to Text）と TTS（Text to Speech）の両方を提供します。

📋 リソース作成手順

  1. Azure Portal の検索バーで「Speech」と入力してクリック
  2. 「+ 作成」をクリック
  3. 以下を設定：
リソースグループ:  既存のもの（例: Hackathon2026）
リージョン:        Japan East
名前:              任意（例: Media-Bot-test）
価格レベル:        F0（無料 — 5時間/月まで）
  1. 「確認と作成」→「作成」をクリック
  2. デプロイ完了後、リソースを開く
  3. 左メニュー「リソース管理」→「キーとエンドポイント」
  4. 「キーの表示」→ キー 1 をコピー
💡 リージョン名（例: japaneast）も控えておきます。appsettings.json の SpeechRegion に使います。

7ビルドとよくあるエラーの修正

📋 ビルドコマンド

cd MediaBot
dotnet build

⚠️ エラー: BadImageFormatException（x86/x64 不一致）

Media SDK のネイティブ DLL が x64 専用のため、プロセスが x86 で起動すると発生します。

対処: MediaBot.csproj の <PropertyGroup> に以下を追加：

<PlatformTarget>x64</PlatformTarget>

⚠️ エラー: Microsoft.Graph バージョン競合

Microsoft.Graph 5.x は Graph.Core 3.x に依存しますが、Communications SDK は Graph.Core 2.x を要求するため競合します。

対処: Microsoft.Graph のバージョンを 4.54.0 に固定します。

⚠️ エラー: IRequestAuthenticationProvider の実装ミス

ValidateInboundRequestAsync の戻り値型は Task ではなく Task<RequestValidationResult> です。

// 正しい実装
public Task<RequestValidationResult> ValidateInboundRequestAsync(HttpRequestMessage request)
{
    return Task.FromResult(new RequestValidationResult { IsValid = true });
}

⚠️ エラー: CreateMediaSession の呼び出し曖昧性

第3引数に null を渡すと、IEnumerable<VideoSocketSettings> と VideoSocketSettings のオーバーロードが曖昧になります。

// 型付き変数で明示的に指定する
IEnumerable<VideoSocketSettings>? noVideo = null;
var mediaSession = _client.CreateMediaSession(
    audioSettings, noVideo, null, null, Guid.NewGuid());

✅ クリーンビルドの確認

dotnet build
# → 0 個の警告、0 エラー が表示されれば完了

8Speech SDK 単体テスト（マイク → STT）

🎯 この手順の目的

Teams なしで Azure Speech SDK の文字起こしが正しく動作することを確認します。Teams から届く音声は PCM 16kHz のバイト列で、このテストでは同じ形式の音声をマイクから取得して動作を検証します。

📋 実行手順

PowerShell（管理者不要） で実行します（dotnet run ではなく exe を直接実行する点に注意）：

cd MediaBot
dotnet build
.\bin\Debug\net48\MediaBot.exe --speech-test
⚠️ dotnet run ではなく .\bin\Debug\net48\MediaBot.exe を使ってください。dotnet run は appsettings.json の読み込みパスが変わります。また、PowerShell から実行することで Enter キー問題も発生しません。

✅ 期待される出力

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
🎤 モード A: マイクテスト
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
マイクに向かって日本語で話してください。
Enter を押すと停止します。

[認識中] こんにちは
[確定 ✅] こんにちは、今日はいい天気ですね。

「認識中」→「確定 ✅」の流れで文字起こしが表示されれば成功です。

💡 Bluetooth イヤホン（AirPods 等）をマイクとして使う場合は、Windows の「サウンドの設定」→「録音デバイス」でデフォルトに設定してください。

🔗 このテストと本番コードの対応

このテスト（マイク → Speech SDK） AudioConfig.FromDefaultMicrophoneInput() ↓ 唯一の違い 本番（Teams → PushAudioInputStream） CallHandler.OnAudioMediaReceived() → Marshal.Copy(buffer.Data, data, ...) → PushAudioInputStream.Write(data) ← ここだけ異なる Speech SDK の認識ロジックは完全に同一

9双方向会話テスト（STT → GPT-4o → TTS）

🎯 この手順の目的

Teams ボットのコア処理（音声認識 → AI 応答生成 → 音声合成）をローカル環境で完全に動作確認します。これが動けば、Teams 接続部分（インフラ）を整えるだけで本番ボットになります。

📋 前提条件

  • appsettings.json の SpeechKey と AzureOpenAiKey が設定済みであること
  • GPT-4o モデルが Azure AI Foundry にデプロイ済みであること

📋 実行手順

.\bin\Debug\net48\MediaBot.exe --chat-test

✅ 期待される動作

  1. 「[あなた] 話してください...」と表示される
  2. マイクに向かって話す（1文で止まると自動で次のステップへ）
  3. 「[あなた] （認識されたテキスト）」が表示される
  4. 「[ボット] 考え中...」→ GPT-4o の応答テキストが表示される
  5. イヤホン / スピーカーから AI の声（日本語女性音声 ja-JP-NanamiNeural）が流れてくる
  6. 「話してください...」に戻り、ループが続く
✅ 音声で話しかけると AI が音声で返してくれる状態が、Teams ボットの会話コアです。

🔗 本番との違い

処理このテストTeams 本番
音声入力FromDefaultMicrophoneInput()（マイク）PushAudioInputStream（Teams 音声）
AI 応答CallGptAsync() ← 同じコードCallGptAsync() ← 同じコード
音声出力FromDefaultSpeakerOutput()（スピーカー）AudioSocket への PCM 送信（要実装）

10本番環境（Azure Windows VM）へのデプロイ概要

🎯 Teams 接続に必要な追加作業

STEP 9 までで会話コアが動作確認できました。本物の Teams 会議に参加させるには以下が必要です：

作業場所難易度
Azure AD アプリ登録（Calls.AccessMedia.All 等の権限）Azure Portal★★☆
Azure Windows VM の作成（B2s 以上、パブリック IP 付き）Azure Portal★★☆
UDP ポート 13016 の開放（NSG 設定）Azure Portal★☆☆
appsettings.json の BotBaseUrl を VM の FQDN に設定コード★☆☆

🌐 ローカルでテストできない理由（UDP の壁）

Teams │ ├─ ① シグナリング（HTTPS）→ Dev Tunnel / ngrok で届く ✅ │ └─ ② メディア（SRTP / UDP）→ ルーター NAT 越えが必要 ❌ Teams が直接 UDP でパケットを送ってくる Dev Tunnel は HTTP のみ対応のため転送できない
💡 テキストチャットボット（HTTP のみ）は Dev Tunnel で動きますが、音声ボット（UDP が必要）は公開 IP を持つサーバーが必須です。これが会社の実装が Azure Windows VM を使う理由です。

  3. 以下を設定して「登録」：
名前:             MediaBot（任意）
サポートされるアカウントの種類: この組織ディレクトリのみ
リダイレクト URI:  （空白でOK）
  1. 登録後、「概要」ページの以下をメモ（後で appsettings.json に入力）：
アプリケーション（クライアント）ID  → AzureAdAppId
ディレクトリ（テナント）ID          → AzureAdTenantId

📋 API アクセス許可の追加

左メニュー「API のアクセス許可」→「+ アクセス許可の追加」→「Microsoft Graph」→「アプリケーションの許可」で以下を追加：

権限名用途

📋 クライアントシークレットの作成

  1. 左メニュー「証明書とシークレット」→「+ 新しいクライアントシークレット」
  2. 説明: 任意（例: MediaBot-2026）、有効期限: 24か月

12Azure Windows VM の作成

🎯 この手順の目的

Media SDK は Windows 専用の DLL を使うため、実行環境として Azure Windows VM が必要です。Teams からの UDP 音声パケットを直接受信するために、パブリック IP アドレス付きの VM を作成します。

📋 VM 作成手順

  1. Azure Portal → 「仮想マシン」→「+ 作成」→「Azure 仮想マシン」
  2. 「基本」タブで以下を設定：
リソースグループ: 既存のもの（例: Hackathon2026）
仮想マシン名:     Media-Bot（任意）
リージョン:        Japan East（東日本）
可用性オプション: インフラストラクチャ冗長は必要ありません
イメージ:          Windows Server 2025 Datacenter Azure Edition（x64、Gen2）
サイズ:            Standard_D2s_v3（2 vCPU / 8 GB RAM）
⚠️ サイズ選択の注意点: B2s_v2 はクォータ不足（0）で作成できない場合があります。Standard_D2s_v3 を選んでください。また、イメージはデフォルトが Ubuntu になっている場合があるので必ず Windows Server を選択します。

📋 ネットワーク設定

  1. 「ネットワーク」タブを開く
  2. 「仮想ネットワーク」が空の場合は「新規作成」リンクをクリックして新規 VNet を作成
  3. 「パブリック IP」は「新規作成」のまま（自動作成される）
  4. 「受信ポートを選択」で RDP（3389）が含まれていることを確認
💡 セキュリティ（NSG の IP 制限）は後で設定できます。まずは動作確認を優先して「パブリック」で作成します。

📋 作成完了の確認

  1. 「確認と作成」タブで内容を確認 → 「作成」
  2. デプロイ完了後（約3〜5分）、VM のリソースページへ移動
  3. 「パブリック IP アドレス」をメモする（例: 40.81.201.8）
✅ VM が「実行中」になれば完了です。

13NSG 設定（Teams メディアポート UDP 13016 開放）

🎯 この手順の目的

Teams は音声（メディア）データを SRTP / UDP で送信します。Bot が音声を受信するには UDP ポート 13016 を NSG（ネットワークセキュリティグループ = AWS の Security Group 相当）で開放する必要があります。

Teams（クラウド側） │ │ SRTP / UDP ポート 13016 ▼ NSG（ファイアウォール）← ここで許可が必要 │ ▼ Azure VM（MediaBot プロセス）

📋 受信セキュリティ規則の追加

  1. VM のリソースページ → 左メニュー「ネットワーク」→「ネットワーク設定」
  2. 「受信ポートの規則を追加する」をクリック
  3. 以下を設定して「追加」：
ソース:            Any
ソースポート範囲:   *
宛先:              Any
宛先ポート範囲:    13016
プロトコル:        UDP
アクション:        許可
優先度:            330（既存ルールと被らない値）
名前:              MediaAudio
⚠️ このポートは Teams の Graph Communications SDK（Media SDK）が使用します。開放しないと Bot が音声を受信できず、会議参加はできても無音のまま処理されます。

💡 NSG とは

概念AzureAWS
ファイアウォールNSG（Network Security Group）Security Group
適用対象サブネット or NIC（ネットワークアダプタ）インスタンス
ルール評価優先度の数字が小さいほど先に評価全許可ルールが合算

14パブリック IP への DNS 名設定

🎯 この手順の目的

Teams の Webhook コールバックに使う BotBaseUrl は固定の URL が必要です。VM に DNS 名（FQDN）を設定することで、IP アドレスが変わっても同じ URL でアクセスできます。また、TLS 証明書の取得にも DNS 名が必要です。

📋 DNS 名の設定手順

  1. VM のリソースページ → 「パブリック IP アドレス」リンクをクリック
  2. 左メニュー「構成」を選択
  3. 「DNS 名ラベル（オプション）」に任意の名前を入力（例: media-bot-2026）
  4. 「保存」をクリック
設定後の FQDN 例:
  media-bot-2026.japaneast.cloudapp.azure.com

これが BotBaseUrl の値になる:
  "BotBaseUrl": "https://media-bot-2026.japaneast.cloudapp.azure.com"
💡 DNS 名ラベルはグローバルに一意である必要があります。他の人が使っている名前は使えません。

15RDP 接続とドライブ共有の設定

🎯 この手順の目的

ローカル PC でビルドした MediaBot の実行ファイル（.exe と DLL 群）を VM に転送するために、RDP のドライブ共有機能を使います。VM 側に .NET SDK をインストールしてビルドするより、ローカルでビルド済みのバイナリをコピーする方が確実で速いです。

📋 VM への RDP 接続

  1. VM のリソースページ → 「接続」→「RDP ファイルのダウンロード」
  2. ダウンロードした .rdp ファイルを開く
  3. 接続ダイアログで VM 作成時に設定した管理者ユーザー名・パスワードを入力して接続
💡 接続時に「証明書エラー」が表示されることがあります。テスト用 VM なので「はい」で接続を続けてください。

📋 ドライブ共有の設定（ファイルコピーを可能にする）

RDP セッションからローカル PC のドライブにアクセスするには、接続前にドライブ共有を有効にする必要があります：

  1. RDP ファイルをダブルクリックして接続ダイアログを開く
  2. 「オプションの表示」をクリック
  3. 「ローカル リソース」タブを選択
  4. 「ローカル デバイスとリソース」セクションの「詳細...」をクリック
  5. 「ドライブ」にチェックを入れて「OK」
  6. 「接続」で RDP セッションを開始
✅ 設定後、VM 内の File Explorer → 「This PC」に「C on [ローカル PC 名]」のようなドライブが表示されます。

16ビルド済みバイナリの VM へのコピー

🎯 この手順の目的

ローカル PC でビルドした net48\ フォルダ（MediaBot.exe + 全 DLL）を VM にコピーします。VM 上で .NET SDK のインストールや dotnet build は不要で、.NET Framework 4.8 は Windows Server 2025 に標準搭載されています。

📋 コピー元フォルダ（ローカル PC）

C:\Users\（ユーザー名）\Hackathon202604\MediaBot\bin\Debug\net48\

このフォルダに含まれる主要ファイル：

ファイル役割
MediaBot.exeエントリポイント（実行ファイル）
MediaBot.exe.config.NET Framework 設定
Microsoft.Skype.Bots.Media.dllTeams 音声処理ネイティブ DLL（Windows x64 専用）
Microsoft.Graph.Communications.*.dllGraph Communications SDK
Microsoft.CognitiveServices.Speech.core.dllAzure Speech SDK ネイティブ DLL
その他の *.dllNuGet パッケージの依存ライブラリ

📋 ファイルコピー手順（RDP ドライブ共有経由）

  1. RDP セッション内で File Explorer を開く（スタートメニュー → File Explorer）
  2. 左サイドバー「This PC」をクリック
  3. 「C on [ローカル PC 名]」ドライブを開く（ドライブ共有で接続したもの）
  4. Users\（ユーザー名）\Hackathon202604\MediaBot\bin\Debug\net48\ フォルダを開く
  5. net48 フォルダを選択して Ctrl+C（コピー）
  6. VM の C:\ に移動して Ctrl+V（貼り付け）
  7. フォルダ名を MediaBot にリネーム（任意）
✅ コピー完了後、VM の C:\MediaBot\MediaBot.exe が存在すれば準備完了です。

📋 appsettings.json の作成（VM 上）

VM 上の C:\MediaBot\ フォルダに appsettings.json を作成します（ローカルのものをコピーしてもOK、ただし BotBaseUrl は変更が必要）：

{
  "AppSettings": {
    "AzureAdTenantId":        "（STEP 11 でメモしたテナント ID）",
    "AzureAdAppId":           "（STEP 11 でメモしたアプリ ID）",
    "AzureAdAppSecret":       "（STEP 11 で作成したシークレット値）",
    "BotBaseUrl":             "https://（STEP 14 で設定した FQDN）",

    "SpeechRegion":           "japaneast",

    "AzureOpenAiEndpoint":    "https://（リソース名）.openai.azure.com/",
    "AzureOpenAiDeployment":  "gpt-4o",
    "CertificateThumbprint":  "（TLS 証明書のサムプリント）"
  }
}
⚠️ BotBaseUrl はローカル用の設定と異なります。STEP 14 で設定した VM の FQDN（https://xxx.japaneast.cloudapp.azure.com）を入力してください。
