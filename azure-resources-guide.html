🏗️IaaS / PaaS の基本概念

IaaS

🖥️ IaaS（インフラのみ）

Infrastructure as a Service

土地だけ借りて、家を自分で建てるイメージ。OSのインストールからセキュリティ設定まで全て自分で管理する。

  • 自由度が高い
  • 管理コストが高い
  • 例：Azure VM
PaaS

🏢 PaaS（プラットフォーム）

Platform as a Service

家具付きマンションを借りるイメージ。インフラはAzureが管理し、開発者はコードを書くことに集中できる。

  • 管理コストが低い
  • スケールしやすい
  • 例：App Service / Container Apps
SaaS

☁️ SaaS（ソフトウェア）

Software as a Service

完成したサービスをそのまま使うイメージ。インフラもアプリも全てプロバイダーが管理する。

  • すぐ使える
  • カスタマイズは限定的
  • 例：Microsoft Teams

⚙️実行基盤の比較

IaaS

🖥️ Azure VM（Virtual Machine）

仮想マシンをそのまま借りるサービス。OSから自分でセットアップする。

  • OSの選択が自由（Windows / Linux）
  • 特殊なライブラリも動かせる
  • 常時起動のためコストが継続発生
  • セキュリティパッチも自分で当てる
  • 現代のBot開発では使うケースは少ない
PaaS

🌐 Azure App Service

Webアプリ・APIをすぐにホストできるフルマネージドサービス。コードをデプロイするだけで動く。

  • Python / Node.js / Java など対応
  • GitやZIPからワンクリックデプロイ
  • SSL・カスタムドメイン対応
  • 自動スケール設定可能
  • Botのバックエンドに最もよく使われる
PaaS+

📦 Azure Container Apps

Dockerコンテナをそのままデプロイできるサービス。マイクロサービス向け。

  • コンテナイメージをそのまま動かせる
  • ゼロスケール対応（使わないときは無料）
  • 複数サービスの連携が得意
  • Kubernetes不要で使いやすい
  • 最近のBot開発で増えているトレンド
比較項目 VM App Service Container Apps
セットアップの手間 大きい 小さい 小さい
コスト（開発時） 常時発生 プラン次第 使った分だけ
スケール 手動 設定が必要 自動
自由度 高い 中程度 中程度
管理コスト 高い 低い 低い
Bot開発での利用頻度 少ない 多い 増加中

🤖Teams Bot関連サービス

PaaS

🔗 Azure Bot Service

TeamsとBotバックエンドを繋ぐ「橋渡し」サービス。Botの登録・認証・ルーティングを担当する。

  • Teams / Slack / Webなど複数チャネルに対応
  • Microsoft App IDの発行と管理
  • メッセージのルーティング処理
  • Bot Framework SDKと連携
  • Teamsボット開発では必須のサービス
💡 「ボットそのもの」ではなく、TeamsとBotを繋ぐ「受付窓口」のイメージ。
PaaS

📞 ACS（Azure Communication Services）

音声・ビデオ・チャット・SMSなどのリアルタイム通信機能をアプリに追加できるサービス。

  • Teams会議への参加・録音が可能
  • 音声のリアルタイム文字起こし
  • Teams Interop（Teamsとの相互接続）
  • Call Automationで通話を自動処理
  • 会議内容を学習するBotに最適
💡 ACSをTeams会議に「参加者」として入室させ、音声をキャプチャして文字起こしする使い方が一般的。

🔄 Bot Service と ACS の役割分担

Azure Bot Service：チャット上のメッセージのやり取りを処理（テキストベースの会話）

ACS：会議・音声・ビデオなどのリアルタイム通信を処理（音声→テキスト変換を含む）

両方を組み合わせることで、「テキストチャット」と「会議音声の学習」を同時に実現できる。

🧠AI・データ系サービス

PaaS

✨ Azure OpenAI Service

OpenAIのGPT-4oなどのモデルをAzureのセキュリティ基盤上で使えるサービス。

  • GPT-4o / GPT-4o mini などのモデルを利用可能
  • 企業向けセキュリティ・プライバシー対応
  • APIキーで簡単に呼び出し可能
  • Azure Foundryから管理・デプロイ
PaaS

🗄️ Azure Cosmos DB

フルマネージドのNoSQLデータベース。会話履歴や学習データの保存に適している。

  • JSONドキュメントをそのまま保存
  • グローバル分散対応
  • 自動スケール
  • Botの会話履歴・ユーザー情報の保存に最適
PaaS

🔍 Azure AI Speech

音声をテキストに変換（STT）、またはテキストを音声に変換（TTS）するサービス。

  • 日本語対応のリアルタイム音声認識
  • 会議の文字起こしに活用
  • テキスト読み上げ（音声案内ボットに活用）
  • ACSと組み合わせて使うケースが多い

🛠️開発ツール：Microsoft 365 Agents Toolkit

📌 Toolkitとは？

VS Codeの拡張機能。Azureリソースそのものではなく、Teams Botの開発・テスト・デプロイを効率化する開発補助ツール。

💻 ローカル開発フェーズ

  • F5 でローカルサーバーを起動
  • Playgroundでチャットをシミュレート
  • Azureなしで素早く動作確認
  • コードを変えてすぐ試せる

☁️ Azureデプロイフェーズ

  • Azure Bot Serviceを自動登録
  • App Service / Container Appsに自動デプロイ
  • Teams App Manifestを自動生成
  • 本物のTeamsで動作確認できる
【開発の流れ】 コード編集 ↓ F5 → Playground で確認（ローカル・無料・速い） ↓ 問題なければ Provision → Deploy（Azureへ） ↓ 本物のTeamsで動作確認 ↓ また編集 → F5 → ...（繰り返し）

🏛️Teams Bot全体アーキテクチャ

【Teams Bot（テキストチャット）の基本構成】 Teams（ユーザーがメッセージ送信） ↓ Azure Bot Service（受付・認証・ルーティング） ↓ App Service / Container Apps（ボット本体のコード） ↓ Azure OpenAI（GPT-4oで返答生成） ↓ Azure Cosmos DB（会話履歴の保存） ↓ Teams（ユーザーへ返答） 【Teams会議学習ボットの構成（ACS追加版）】 Teams会議（音声） ↓ ACS（会議参加・音声キャプチャ） ↓ Azure AI Speech（音声→テキスト変換） ↓ Azure OpenAI（内容を要約・学習） ↓ Azure Cosmos DB（学習内容を保存） ↑ Azure Bot Service ← Teams チャット ↓ 「この会議でこんな話が出ていましたよ」と返答
